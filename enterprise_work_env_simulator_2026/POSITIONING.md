# Positioning — Primary: Pre-Customer Product Validation

*Updated 2026-05-28*

## One sentence

**Synthetic customers try your product inside a fake company — before you have real users.**

## The tool being evaluated

**Always the builder’s product** (web app, API product with UI, internal tool with UI).

The synthetic stack (Slack, Jira, CRM, email) is **context**, not the subject of the eval — unless you explicitly sell “integration rehearsal” as a module.

## Primary ICP

| | |
|---|---|
| **Who** | Founder, PM, design engineer, solo builder |
| **Stage** | Pre-launch → early beta (<50 real users) |
| **Company** | B2B SaaS, dev tools, workflow products |
| **Trigger** | “We’re about to launch / open beta and we’re guessing” |

## Secondary ICP (later GTM)

B2B SaaS **Solutions Engineering** — same engine, message = “rehearse prospect POC.”

## Jobs to be done

1. **End-to-end** rehearsal — full journeys, not page spot-checks  
2. **Every user type** — replicate flows across personas (new, power, admin, returner, heavy user)  
3. **Human-ready quality** — information, speed, UI/UX, functionality, adaptability  
4. **After repeated use** — gaps and frustrations that show up on the 10th or 100th session  
5. **Positives too** — delights, impressive moments, time saved (not bugs-only)  
6. Integrations + compliance **signals** in context  
7. **Compare** runs after each release (regression rehearsal)

See `EVALUATION_FRAMEWORK.md` for the full rubric.

## Not claiming

- Replacement for all user research  
- SOC2 certification  
- Proof of product-market fit (only product *readiness*)
