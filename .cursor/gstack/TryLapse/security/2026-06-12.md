- 10 new REST endpoints (persona library CRUD, recordings, crawl-graph, progress, credentials, run control)

## Findings

### P1 — Critical

#### P1-01: Path Traversal — 7 endpoints share the same bug

**Files:** `server.py:282`, `server.py:432`, `server.py:1203-1211`, `server.py:1218`, `server.py:1226-1235`, `server.py:1246-1254`, `server.py:1546-1550`  
**Also via:** `recording.py:17`, `recording.py:31`, `recording.py:42`, `progress.py:65`, `progress.py:187`

Every endpoint that accepts a `run_id` or `journey_id` from the URL path or query string constructs a file path by direct concatenation without a bounds check:

```python
# server.py:282 — crawl-graph
run_id = path.strip("/").split("/")[2]
graph_path = root / "runs" / f"{run_id}-crawl-graph.json"  # NO BOUNDS CHECK

# progress.py:65 — progress polling
self._path = artifacts_root / "runs" / f"{run_id}-progress.json"

# recording.py:17 — save recordings
recording_dir = artifacts_root / "artifacts" / run_id / "recordings"
```

**Exploit path:**
- `run_id = "../personas"` → reads `artifacts_root/personas.json` (full persona library, includes imported credentials metadata)
- `run_id = "../users"` → attempt to read user DB
- `run_id = "../../some_target"` → reads up to filesystem root for `.json`/`-progress.json`-suffixed files
- POST `runId = "../../injected"` → creates directories or writes data outside the artifacts tree

**Why it's P1:** The recording save endpoint writes to a path derived from user input. An attacker with network access to the server (not just localhost — consider Vercel or any cloud deployment) can write files to arbitrary locations on the server filesystem.

**Fix — apply to every affected site:**
```python
def _safe_run_path(root: Path, *parts: str, suffix: str = "") -> Path | None:
    """Return resolved path only if it stays within root, else None."""
    # Reject obviously malicious inputs fast
    for part in parts:
        if "/" in part or "\\" in part or ".." in part:
            return None
    resolved = (root.joinpath(*parts) if not suffix
                else root.joinpath(*parts[:-1]) / f"{parts[-1]}{suffix}").resolve()
    return resolved if resolved.is_relative_to(root.resolve()) else None
```

Then at each call site:
```python
# crawl-graph
graph_path = _safe_run_path(root, "runs", run_id, suffix="-crawl-graph.json")
if graph_path is None:
    self._send_json({"error": "invalid run_id"}, status=400)
    return
```

The same helper covers progress, recordings, control signals, and the persona-library import.

### P2 — High

#### P2-01: Cross-Workspace Data Leak — inconsistent config_prefix extraction

**Files:** `server.py:320-343` (`/api/summaries`), `server.py:447-467` (`/api/trends`)  
vs `server.py:575-600` (`/api/jobs`)

The jobs endpoint extracts the full product slug via regex:
```python
m = _re.match(r"^(.*)-\d{8}-\d{6}$", stem)
config_prefix_q = m.group(1) if m else stem  # → "faculty-dashboard-eight-vercel-app"
```

But summaries and trends only take the first hyphen segment:
```python
config_prefix = _P(cp).stem.split("-")[0]  # → "faculty" ← WRONG
```

**Impact:** Two workspaces with configs `argyle-hr-20260611-052339.yaml` and `argyle-sales-20260612-090000.yaml` get identical `config_prefix = "argyle"` and see each other's run summaries and trend data. This is a cross-tenant read.

**Fix:** Replace the `split("-")[0]` pattern in both endpoints with the same regex used in `/api/jobs`:
```python
import re as _re
stem = _P(cp).stem
m = _re.match(r"^(.*)-\d{8}-\d{6}$", stem)
config_prefix = m.group(1) if m else stem
```

#### P2-02: API Key Race Condition via Shared `os.environ`

**File:** `server.py:850-917` (`/api/product/analyze`)

When `llmApiKey` or `visionApiKey` are passed in the request body, they are temporarily set in the process-wide `os.environ`:
```python
_os.environ["REHEARSE_LLM_API_KEY"] = llm_api_key  # shared namespace
# ... 3-minute crawl + vision + LLM call happens here ...
finally:
    _os.environ["REHEARSE_LLM_API_KEY"] = _orig_llm  # restored after
```

The server uses `ThreadPoolExecutor` for enqueuing runs (`jobs.py`). During the 3-minute crawl window, a concurrent job thread that reads `REHEARSE_LLM_API_KEY` from `os.environ` will pick up User A's key instead of the configured key. On a multi-user deployment this leaks User A's API key to User B's job.

**Fix:** Pass the key as an explicit parameter through the call chain rather than via `os.environ`. If that's too invasive short-term, use a per-request context object (threading.local) or lock the environment during the mutation window:
```python
with _api_key_lock:  # threading.Lock at module level
    _os.environ["REHEARSE_LLM_API_KEY"] = llm_api_key
    try:
        result = run_analysis(...)
    finally:
        _os.environ["REHEARSE_LLM_API_KEY"] = _orig_llm or ""
```
Note: this is a band-aid. The right fix is explicit parameter passing.

#### P2-03: `configPrefix` Query Param Bypasses Workspace Isolation in `/api/jobs`

**File:** `server.py:560-563`

```python
config_prefix_q = (qs.get("configPrefix") or [""])[0].strip()
if not config_prefix_q:
    payload = self._require_jwt()
    ...  # derive prefix from JWT workspace
```

If `configPrefix` is provided in the query string, it takes precedence over the JWT-derived workspace without any ownership validation. An authenticated user from workspace "argyle" can query `GET /api/jobs?configPrefix=faculty-dashboard` and read another workspace's jobs.

**Fix:** Ignore the `configPrefix` query param if the user is authenticated and their JWT-derived prefix differs, or remove the query-param override entirely (only accept it from an internal server-side context):
```python
# If authenticated, always derive prefix from JWT; ignore query param
if payload:
    config_prefix_q = <derive from JWT workspace>
else:
    config_prefix_q = (qs.get("configPrefix") or [""])[0].strip()
```

### P3 — Medium

#### P3-01: `/api/credentials` GET is Unauthenticated

**File:** `server.py:238-244` (placed before `_check_auth` call at line 270)

Returns `{hasEmail: bool, hasPassword: bool}` with zero auth. This reveals server configuration state to anyone who can reach the port, including same-machine processes and LAN users.

**Fix:** Move the handler to after the `_check_auth` gate, or add it to `_PUBLIC_PATHS` explicitly with a comment explaining the intent. The current omission looks like an oversight.

#### P3-02: No Input Validation on `POST /api/persona-library`

**File:** `server.py:633-634`, `persona_store.py:save_persona()`

The raw request body dict is passed directly to `save_persona()`. No field schema is enforced. Attacker can inject arbitrary keys into `personas.json` that will be returned on every `GET /api/persona-library` response (potentially inflating response size or poisoning downstream consumers).

**Fix:** Whitelist accepted fields in the save handler:
```python
ALLOWED_PERSONA_FIELDS = {
    "id", "name", "role", "goals", "enabled", "tech_literacy",
    "patience", "trust_level", "character", "usage_context", "tags",
    "source", "created_at", "updated_at",
}
body = {k: v for k, v in body.items() if k in ALLOWED_PERSONA_FIELDS}
saved = save_persona(root, body)
```

#### P3-03: Plaintext Credential Persistence in `.env`

**File:** `server.py:_write_cred_env()` (lines 63-89), called from `server.py:860` and `server.py:1512`

User credentials (email + password for the _target product_, not LR itself) are written to `{project_root}/.env` in cleartext. This is intentional (runner subprocesses read them via `load_dotenv`), but:
- `.env` is in `.gitignore` ✓
- Any plaintext write carries shoulder-surfing and log-exposure risk
- If a user accidentally adds `artifacts/` or `*.env` to a deploy artifact, credentials ship

**Fix (short-term):** Add a warning in the credentials panel that these are stored in cleartext on disk at `launch-rehearsal/.env`.  
**Fix (long-term before GA):** Use macOS Keychain (via `keyring` library) for credential storage; keep `.env` as a fallback for non-interactive headless mode.

---

#### P3-04: JWT Secret Ephemeral on Server Restart

**File:** `auth_store.py:31`

```python
_JWT_SECRET: str = os.environ.get("REHEARSE_JWT_SECRET") or secrets.token_hex(32)
```

If `REHEARSE_JWT_SECRET` is not set (default for new installs), the secret regenerates on every server restart. All existing sessions are invalidated. This is annoying but not a security bug per se.

**Impact:** Users lose sessions after every restart (which happens after every Python change per the workflow). They re-login, new JWT issued — fine.

**Note:** Becomes a real issue in multi-process or load-balanced deployments where each instance has a different secret. For now acceptable; must set `REHEARSE_JWT_SECRET` before horizontal scaling.
