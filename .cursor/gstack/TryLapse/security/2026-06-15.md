## New Findings ‚Äî This Review

### P2-NEW-01 ¬∑ configPath filesystem traversal in job-enqueue endpoints [FIXED]

**Endpoints:** `POST /api/jobs`, `POST /api/jobs/cohort`, `POST /api/jobs/variant`  
**Lines (pre-fix):** server.py:1530, 1547-1548, 1566

**Issue:** All three endpoints accepted a caller-supplied filesystem path (`configPath`, `configAPath`, `configBPath`) and only checked `cfg.is_file()` ‚Äî no bounds check. An authenticated user could supply `/etc/passwd`, `../../.env`, or any world-readable file. The file would be YAML-parsed by the job executor; the parsed content leaks in error messages and logs.

**Fix applied:** `cfg.resolve().relative_to(root.resolve())` guard before `is_file()` check. If the resolved path escapes `artifacts_root`, returns HTTP 400 immediately. Uses `pathlib.relative_to()` rather than string prefix comparison to avoid `/foo/bar` vs `/foo/bar-evil` ambiguity.

### P2-NEW-02 ¬∑ `.env` newline injection via POST /api/credentials [FIXED]

**Location:** `_write_cred_env()` ‚Äî server.py:98  
**Lines (pre-fix):** 113-116

**Issue:** `email` and `password` were written directly into `.env` as `KEY=VALUE\n` lines without stripping embedded newlines. An attacker supplying `email="legit@example.com\nSECRET_KEY=injected"` would insert an extra line into the project `.env` file, allowing arbitrary environment-variable injection on next server restart.

**Fix applied:** `.replace("\n", "").replace("\r", "")` applied to both values before any use.

### P2-NEW-03 ¬∑ Missing `_safe_id` on three endpoints [FIXED]

**Issue:** Three POST/GET/DELETE handlers extracted IDs directly from the URL path without calling `_safe_id()`, bypassing the path-traversal guard applied consistently elsewhere.

| Endpoint | Variable | Pre-fix line | Fix |
|----------|----------|--------------|-----|
| `POST /api/runs/{runId}/chat` | `run_id = parts[2]` | 1484 | `_safe_id(run_id, "runId")` before `load_bundle()` |
| `POST /api/annotations/{runId}` | `run_id = path.split("/")[-1]` | 1586 | `_safe_id(run_id, "runId")` before `save_annotation()` |
| `GET /api/persona-library/{id}` | `pid = path.split("/")[-1]` | 593 | `_safe_id(pid, "id")` before `get_persona()` |
| `DELETE /api/persona-library/{id}` | `pid = path.split("/")[-1]` | 1656 | `_safe_id(pid, "id")` before `delete_persona()` |

### P2-NEW-04 ¬∑ No rate limiting on expensive LLM + Playwright endpoints [FIXED]

**Endpoints (pre-fix):** `/api/product/analyze`, `/api/persona-library/generate`, `/api/journeys/draft`, `/api/personas/draft`, `/api/personas/suggest`

**Issue:** The existing rate limiter only covered `/api/jobs*`. The above endpoints each trigger at minimum one LLM call (some also launch Playwright), making them the most expensive operations per request. Without a rate limit, a single IP can spam these indefinitely ‚Äî a direct cost-abuse vector once the Railway deployment is live.

**Fix applied:**
- `rate_limit.py`: Added `"llm": 10` bucket (10 requests/60s) and `_LLM_PATHS` set mapping to it
- `server.py`: Existing rate-limit check extended to `path in _LLM_PATHS` in addition to `/api/jobs*`

### P3-NEW-01 ¬∑ `X-Forwarded-For` rate-limit bypass (open finding)

**Location:** server.py line ~765 (unchanged)

**Issue:** `client_ip = self.headers.get("X-Forwarded-For", self.client_address[0]).split(",")[0].strip()` ‚Äî any caller can set an arbitrary `X-Forwarded-For` header and rotate it to bypass rate limiting. On Railway behind a reverse proxy the header is injected by the CDN, but a direct connection to the container port bypasses this.

**Status:** Not fixed in this pass. Correct fix requires knowing Railway's proxy trust chain (typically: trust only the rightmost `X-Forwarded-For` entry, or strip the header entirely and use `self.client_address[0]`). For current self-hosted deployments, the attack surface is low since the API token gate prevents anonymous access. **Track for Railway hardening milestone.**
