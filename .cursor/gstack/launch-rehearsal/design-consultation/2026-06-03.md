# Design Consultation — Launch Rehearsal Dashboard
**Date:** 2026-06-03  
**Category:** B2B SaaS monitoring / pre-launch readiness operator console  
**Stack:** React + TanStack Router + Tailwind v4 + shadcn/ui + Recharts  
**Register:** Product (design serves the task)

---

## Layer 1 — Conventional Wisdom in This Category

The 2026 B2B monitoring dashboard canon is well-established. The dominant aesthetic comes from Linear, Vercel, Stripe, and Raycast:

- **Sidebar 240–280px** with persistent nav + nested groups, flat active state with subtle background fill
- **4–6 KPI cards** in a top row as the metric strip — the "is everything OK?" row
- **One status vocabulary** — green/yellow/red meaning the same thing everywhere (not decorative)
- **Progressive disclosure** — show the minimum for next decision; reveal depth on click
- **Typography is hierarchy** — one well-tuned sans, tight 1.125–1.2 scale ratio, size carries weight
- **Calm default state** — nothing glows, pulses, or animates when everything is fine

What everyone is moving *away from*:
- Pastel gradients, glassmorphism, decorative card borders
- "Hero metrics" with outsized font sizes
- Overcrowded metric strips (>8 KPI cards = decision paralysis)
- Identical border weight on every container (visual flatness)

---

## Layer 2 — Your Specific Wedge

Launch Rehearsal is not a traditional monitoring tool. It's a *pre-launch readiness console* — more like a "test results dashboard" than a "production alert stream." That's an important distinction:

| Monitoring dashboards | LR Command Center |
|-----------------------|-------------------|
| "Is prod healthy right now?" | "Is this ready to ship?" |
| Real-time continuous | Run-by-run episodic |
| Engineers/SRE | Eng leads, QA, founders |
| Red = alert | Red = blocked, not urgent |
| Dense time-series charts | Evidence panels + narrative |

This means you can and should **break from the alert-stream aesthetic** in specific ways:
- **Readiness band** is a verdict, not a live metric — it should feel like a grade, not a pulse
- **Evidence** is the core product surface — screenshots, step sequences — more like a code review than a dashboard
- **Narrative + digest** (the NLU layer already built) is a differentiator: dashboards don't explain, yours does

## Layer 3 — Visual References

### Steal from these

**Linear (linear.app)**  
The density/whitespace balance. Nav groups with quiet separators, not bold dividers. Flat sidebar item active state (`bg-surface-2` fill, no left border). Clean chip vocabulary. JetBrains Mono for technical identifiers. *Most relevant: their issue detail layout — one-column primary info, sparse metadata row.*

**Vercel dashboard**  
Binary visual signals for deployment status. Readiness as a headline verdict. The concept of "deployment passed / failed" framing maps well to your Green/Amber/Red band. Clean monochrome chart baseline with single-color overlay.

**Raycast**  
Command palette (Cmd+K) interaction model. Fast operator scanning. Keyboard-first affordances. The way they handle "type to filter" inside a modal is cleaner than traditional dropdowns.

**Stripe Radar (fraud dashboard)**  
Evidence-bound scoring. Each finding has a "why" panel with linked evidence. Score + band + detail panel is the exact LR information pattern. *Most relevant: how they present a "risk score" without it feeling like a gauge.*

### Avoid these
- **Datadog / Grafana** — density-first, built for SRE, every panel fighting for attention, visually exhausting for a 20-min readout session
- **AdminLTE / TailAdmin templates** — overcarded, too many accent colors, feels like SaaS boilerplate
- **Mixpanel / Amplitude** — chart-heavy, meant for product analytics not evidence browsing

---

## Differentiation Thesis

**The design language should feel like a thoughtful code review, not a live operations center.**

- **Color:** Warm off-white canvas (already there) + cool slate sidebar + single indigo primary. No second accent. Status colors semantic only (green/yellow/red).
- **Typography:** Inter is acceptable for product UI per the product register (familiarity is a feature). The real issue is the *scale* — too many one-off sizes (10px, 11px, 12px, 13px, 14px all co-existing). Lock to 5 sizes: 11 / 13 / 15 / 18 / 24.
- **Evidence panels:** Should feel like GitHub diff / PR review. Monospace for step IDs, clean screenshot thumbnails with context overlays, step timeline as a linear sequence.
- **Readiness band:** Should feel like a test suite result (PASS / WARN / FAIL) — a *verdict* with a number, not a real-time gauge pulsing anxiously. Consider treating Green as the *default quiet state* and only visually activating for Amber/Red.
- **Distinct element:** The dimension rollup grid (8 axes) is unique — no competitor shows a structured rubric. This should be the visual signature of the product. Make each axis card feel like a professional rubric row, not a KPI widget.

## Theme Direction to Follow

### Name: "Evidence Console"

This is a refinement of the existing direction, not a redesign. Changes to make:

| Area | Current | Target |
|------|---------|--------|
| **Font sizes** | 10/11/12/13/14px chaotic spread | Lock to 5-step scale: 11/13/15/18/24 |
| **Border weight** | Equal 1px on all containers | Quiet panels = 1px / 40% opacity; Active panels = 1px / 100% |
| **Status indicators** | Green pulses on healthy state | No pulse/glow when healthy; reserve animation for Amber/Red |
| **Readiness display** | Circular gauge with glow ring | Verdict chip + large numeral, no animation at rest |
| **Dimension cards** | Functional but visually flat | Subtle left-border color stripe per tone (3px colored left-border) |
| **Sidebar groups** | Bold uppercase "MONITOR" labels | Quiet uppercase 10px slate labels (already correct in intent) |
| **Primary CTA** | `bg-primary` filled button | Keep — this is right |
| **Secondary CTAs** | `border border-border` outline | Should be quieter: borderless with `hover:bg-surface-2` |

### What NOT to change
- The oklch color token system — excellent, keep it
- The warm off-white background — distinctive and calm
- Semantic status colors (ready/warn/danger)
- The `panel` utility base style
- JetBrains Mono for IDs and code
